# AnchorScan V1.2 Stable Deploy And Controlled Scan Design

Date: 2026-07-08

## Goal

V1.2 focuses on making AnchorScan easier to deploy on a new machine and safer to operate during scans.

The release should answer three operator questions before work starts:

- Can this machine run AnchorScan correctly?
- What exactly will this scan do?
- Can an existing SQLite database upgrade without manual repair?

## Scope

Included:

- a small release/package workflow for the AnchorScan binary and config files
- stronger `doctor` checks for tool paths, config files, report/data directories, and SQLite access
- one shared preflight path used by CLI and Web Console before starting a scan
- a clear scan summary: targets, ports, profile, workers, selected tool args, NSE config, and nuclei tag config
- a minimal SQLite migration system using a `schema_migrations` table
- tests for packaging assumptions, preflight behavior, and migration idempotency

Not included:

- bundling `rustscan`, `nmap`, `httpx`, or `nuclei`
- login, roles, multi-user permissions, or distributed scanning
- a GUI installer
- a complex background job queue
- major visual redesign beyond the screens needed for preflight/doctor output

## User Experience

### New Machine Setup

The operator should be able to:

1. Build or download the AnchorScan binary.
2. Put `config/` next to the binary or pass `--config`.
3. Run `anchorscan doctor --config config/default.yaml --db data/scans.sqlite --reports reports`.
4. Fix any reported missing tool paths or unwritable directories.
5. Start the Web Console or run a CLI scan.

The package should contain:

- `anchorscan`
- `config/default.yaml`
- `config/nse.yaml`
- `config/service-aliases.yaml`
- `config/service-tags.yaml`
- a short deploy README

### Scan Preflight

Before a scan starts, AnchorScan should produce a preflight result with:

- target count and normalized target list
- selected port expression
- selected scan profile
- resolved host worker count
- configured tool paths and whether each required tool exists
- configured extra args for rustscan, nmap, httpx, and nuclei
- NSE mapping availability
- nuclei tag mapping availability
- report and database writability checks

CLI should print the summary before the first rustscan event.

Web Console should show the same summary on the scan launch path. If the preflight has blocking errors, the scan should not start.

Warnings should not block. Examples:

- `full` port scan selected
- many targets selected with `fast` profile
- nuclei path missing while nuclei checks are configured

Errors should block. Examples:

- no targets
- invalid port expression
- missing rustscan or nmap path
- database cannot be opened
- report output path is not writable

## Architecture

### Package Workflow

Add a minimal build surface:

- `make test`
- `make build`
- `make package`

`make package` should create a local release directory or archive under `dist/`.

No cross-platform release matrix is required for V1.2. The command may build for the current OS/architecture only.

### Doctor

Keep the existing doctor command and expand its checks.

Doctor should remain read-mostly. It may create missing local runtime directories when that matches existing behavior, but it should not rewrite config files or run scans.

### Preflight

Add a small package or app-level helper that accepts the same effective scan inputs used by CLI and Web Console.

It should return structured data:

```go
type PreflightResult struct {
    Summary  ScanSummary
    Warnings []CheckMessage
    Errors   []CheckMessage
}
```

The exact type names can follow the surrounding code. The important boundary is that CLI and Web Console consume the same result instead of duplicating checks.

### SQLite Migrations

Replace ad hoc `ALTER TABLE` startup patches with ordered migrations:

- create `schema_migrations(version INTEGER PRIMARY KEY, name TEXT NOT NULL, applied_at TEXT NOT NULL)`
- apply missing migrations in order inside a transaction
- make migrations idempotent at the database level where practical
- preserve existing data

Existing schema changes should become named migrations. New empty databases should be initialized through the same migration path.

## Error Handling

- Preflight errors stop a scan before any external scanner runs.
- Tool execution failures during a scan keep the current behavior: failed targets are reported, partial successful targets may still produce a report.
- Migration failure should stop DB open and return a clear error.
- Package creation failure should fail fast with the underlying command or file error.

## Testing

Add focused tests:

- fresh DB creates all expected tables and records migrations
- old DB missing newer columns upgrades successfully
- migrations can run twice without changing results
- preflight rejects missing targets and invalid ports
- preflight reports missing required tool paths
- CLI scan help and Web scan launch continue to agree on accepted target/port input
- package workflow includes binary and config files

Manual verification:

```bash
go test ./...
make package
./dist/<package>/anchorscan doctor --config ./dist/<package>/config/default.yaml --db ./dist/<package>/data/scans.sqlite --reports ./dist/<package>/reports
```

## Acceptance Criteria

- A new machine setup path is documented and works from a clean checkout.
- `doctor` gives actionable output for missing tools, bad config paths, DB problems, and report directory problems.
- CLI and Web scans use the same preflight logic.
- Blocking scan issues are caught before rustscan starts.
- SQLite upgrades are deterministic and safe to run repeatedly.
- Existing V1.1 behavior remains intact.
- `go test ./...` passes.

## Deferred

- Bundled third-party scanner binaries
- Signed installers
- Automatic template updates in air-gapped environments
- Full frontend redesign
- Multi-scan scheduling
