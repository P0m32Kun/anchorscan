# AnchorScan v1.4 Design: Filter UX, Audit Artifacts, Rich Report Export

Date: 2026-07-09
Status: approved design draft
Scope: v1.4 optimization phase

## Goals

v1.4 focuses on three user-visible improvements:

1. Replace free-text severity filtering with fixed multi-select severity filters.
2. Persist tool process files during automated pipeline scans for audit and debugging.
3. Make report export a real file download and include useful vulnerability evidence.

This is intentionally a small optimization release. It should reuse the existing scan, report, and deletion flow instead of introducing a full artifact-management subsystem.

## Non-goals

- No PDF export in v1.4.
- No new artifact index table unless plain directory storage becomes insufficient.
- No UI for browsing every artifact file beyond linking/showing the run artifact directory.
- No new external dependencies.

## Approach

Use the smallest complete approach:

- Keep report filtering in the existing report page handlers/templates.
- Add an artifact root/path to scan options and scan run records.
- Write raw or parsed tool outputs to files as each tool runs.
- Add report export endpoints that set download headers.
- Delete artifact directories when their run or project is deleted.

Alternatives considered:

1. Add a `scan_artifacts` table for every generated file. This is useful later, but v1.4 only needs audit files to exist and be cleaned up.
2. Build a general artifact center for reports, temp files, exports, and future files. This is overkill for the current requirement.

## Severity Filter UX

Severity is a fixed enum, so the UI should not ask users to type it.

Supported severities:

- `critical`
- `high`
- `medium`
- `low`
- `info`

The vulnerability list uses checkbox/chip multi-select controls. Multiple selected severities are combined with OR semantics. No selected severity means no severity filter.

Example URL shape:

```text
/reports/<run_id>?severity=critical&severity=high
```

The backend accepts repeated `severity` query params. It should also tolerate comma-separated input if that is cheap, but repeated params are the primary form because they work naturally with HTML checkboxes.

Asset/search views that expose vulnerability severity filters should use the same fixed multi-select control.

## Scan Audit Artifacts

Automated pipeline scans save process files by default.

Default artifact root:

```text
data/artifacts
```

Each scan creates one run directory:

```text
<artifact_root>/<run_id>/
```

Users can override the artifact root from the scan form and CLI. The app always creates the `<run_id>` child directory under the chosen root.

### Stored Path

Add `artifact_dir` to `scan_runs`.

The value stores the exact run artifact directory, not just the root, for example:

```text
data/artifacts/run-20260709-120000.000000000
```

This is needed because users can choose a custom artifact root per scan.

### File Naming

File names must identify the tool and target and must not overwrite each other.

Examples:

```text
nmap-alive-targets.xml
rustscan-172.22.0.4-ports.txt
nmap-service-172.22.0.4-ports-80,445.xml
nse-172.22.0.4-445-smb-protocols.xml
httpx-172.22.0.4-80.jsonl
nuclei-172.22.0.4-80-smb.jsonl
```

Rules:

- Include tool name first.
- Include target IP/host or a safe target label.
- Include port, script, tags, or mode when relevant.
- Sanitize `/`, `:`, whitespace, shell metacharacters, and other unsafe path characters.
- If a sanitized name collides, append a short suffix such as `-2` or a short hash.

### Tool Outputs

Persist these files during automated pipeline scans:

- nmap alive sweep: raw XML from `nmap -sn -oX -`.
- rustscan: raw greppable/open-port output plus the parsed open ports.
- nmap service detection: raw XML from service fingerprinting.
- nmap NSE: raw XML or script output for each NSE execution.
- httpx: raw JSONL output.
- nuclei: raw JSONL output.

If a command fails but returns stdout/stderr, save what is available before returning the error. This makes failed scans auditable.

### Cleanup

Run deletion removes the stored `artifact_dir` after verifying the run is not running.

Project deletion removes artifact directories for all runs in the project.

For safety, deletion only removes the run directory created by the app. It must never remove the user-selected artifact root itself.

## Report Export

Report export should download files instead of navigating to a rendered page.

Add endpoints like:

```text
/reports/<run_id>/export?format=html
/reports/<run_id>/export?format=json
/reports/<run_id>/export?format=csv
```

Responses set:

```text
Content-Disposition: attachment; filename="anchorscan-<run_id>.<ext>"
```

Supported formats in v1.4:

- HTML: human-readable report.
- JSON: structured full report.
- CSV: findings/assets table for spreadsheet review.

PDF is explicitly deferred.

## Rich Vulnerability Evidence

Reports should include enough evidence to support manual verification.

Each finding should show:

- ID
- severity
- source (`nuclei`, `nse`, `manual-review`)
- IP
- port
- target URL or matched target
- summary
- evidence/output

Source-specific evidence:

- nuclei: `matched-at`, `matcher-name`, `extracted-results`, `curl-command`, and raw JSON snippet when available.
- NSE: script output.
- manual review: why the service needs manual verification and what to check.

The existing `report.Finding.Output` field is the primary evidence field for v1.4. Avoid schema churn unless a test proves it is insufficient.

## Data Flow

```text
scan form / CLI
  -> ScanOptions{ArtifactDir or ArtifactRoot}
  -> RunScan creates <artifact_root>/<run_id>/
  -> each tool call writes its process file
  -> fingerprints/findings saved to SQLite
  -> report JSON written as today
  -> report export endpoint rebuilds/loads report and sends attachment
```

## Error Handling

- If artifact directory creation fails, fail the scan before tools run. Silent loss of audit files would violate the v1.4 goal.
- If writing one artifact file fails, fail the scan and record an event with the path and error.
- If no output exists for a tool, do not create an empty artifact unless the file itself communicates a parsed result, such as rustscan open ports.
- If report export format is unknown, return HTTP 400.
- If a report/run does not exist, return HTTP 404.

## Testing

Add focused tests only where behavior can regress.

### Severity filters

- Single severity selection returns only matching findings.
- Multiple severities return the union.
- No selected severity returns all findings.

### Artifact files

- Scan creates `<artifact_root>/<run_id>/`.
- nmap, rustscan, httpx, and nuclei outputs are saved with distinct names.
- Failed tool output is saved before the scan returns an error.
- Deleting a run removes its artifact directory.
- Deleting a project removes artifact directories for its runs.

### Report export

- HTML/JSON/CSV exports return `Content-Disposition: attachment`.
- Exported report includes finding evidence/output.
- Unknown export format returns HTTP 400.

## Implementation Notes

Use existing patterns first:

- Extend `ScanOptions` rather than adding a new scan subsystem.
- Extend `scan_runs` migration/model with `artifact_dir`.
- Keep file writing in small helper functions using the standard library.
- Reuse existing report model and only add fields if required by tests.

The v1.4 implementation is complete when users can run a normal pipeline scan, inspect the generated artifact directory, filter vulnerabilities by fixed severities, download a richer report, and delete the run/project without leaving artifact files behind.
