# 现代化 Web Console 视觉重构

## Problem Statement

AnchorScan Web Console 已具备完整的项目、扫描、报告、工具、知识库和配置流程，但当前视觉实现以黑橙工业风为主，页面层级偏重，模板中存在大量内联样式，高密度报告页的筛选、操作和结果区域缺少统一的信息架构。

当前前端是 Go `html/template`、单个全局 CSS 文件和少量原生 JavaScript，不需要也不适合为了美化迁移到 React、Vue 或新的 UI 框架。代表性本地报告页包含约 108 个交互控件和 580 个渲染后的内联 `style` 属性，说明主要问题是样式接口分散，而不是缺少前端运行时。

## Solution

将所有桌面 Web 页面重构为浅色、现代、简洁的 Apple-inspired 产品界面。视觉参考 macOS 系统设置、iCloud Web 和 Apple Developer 控制台，而不是 Apple 官网营销页。

新的界面保留安全产品需要的信息密度：背景、层级和材质保持轻盈，表格、筛选器和操作区保持紧凑。主色改为克制的系统蓝；红、橙、黄仅承担漏洞等级和告警语义。保留并简化 AnchorScan 盾牌标识，继续使用左侧导航。

实施按阶段推进。先用扫描报告页原型验证视觉方向，再建立共享设计令牌和桌面应用外壳，随后依次迁移报告、项目与扫描、其余控制台页面和独立导出报告。每个阶段必须提供 1440px 截图并经用户确认后，才能进入下一阶段。

## User Stories

1. 作为安全分析人员，我希望报告页视觉安静且层级清楚，以便长时间阅读大量资产和漏洞结果。
2. 作为安全分析人员，我希望筛选、视图切换和主要操作集中且可预测，以便快速缩小报告范围。
3. 作为项目操作者，我希望项目配置、扫描创建和任务状态使用统一的表单与状态语言，以便减少误操作。
4. 作为控制台用户，我希望所有页面使用同一套导航、字体、颜色、间距和交互状态，以便产品看起来完整而非页面拼接。
5. 作为报告交付人员，我希望独立 HTML 报告与 Web Console 使用一致的视觉语言，以便离线交付不显得是另一套产品。
6. 作为键盘用户，我希望所有可操作元素都有可见焦点、清晰标签和足够对比度，以便不依赖鼠标完成主要流程。
7. 作为现有用户，我希望所有路由、数据、筛选语义、扫描能力和下载格式保持不变，以便视觉升级不改变工作流。

## Approved Product Decisions

- 视觉方向：浅色 Apple-inspired 产品界面，排除黑橙工业风。
- 参考类型：macOS 系统设置、iCloud Web、Apple Developer 控制台；不采用营销页式超大标题和过量留白。
- 密度原则：视觉轻盈、操作紧凑；不能为了美观降低报告信息密度。
- 主色：系统蓝；橙色不再承担品牌强调，只保留风险和告警语义。
- 导航：保留左侧边栏，改为更轻、更窄的浅色导航。
- 品牌：保留并简化盾牌标识，不进行完整 Logo 重设计。
- 材质：半透明和模糊只用于侧边栏、吸顶工具栏和浮层；数据表格与内容面保持接近实色。
- 范围：所有现有 Web 页面和独立导出 HTML 报告全部纳入。
- 交互：允许调整筛选分组、吸顶工具栏、固定表头和次要操作收纳；不新增业务能力。
- 平台：桌面专用。1440px 为主要设计基准，最低支持宽度 1280px；低于 1280px 仅保证页面可滚动。
- 主题：本计划完全排除深色模式。
- 验收：每阶段先提供 1440px 关键页面截图，经用户确认后继续。

## Implementation Decisions

- 保留 Go `html/template`、原生 CSS 和原生 JavaScript，不引入前端框架、UI 组件库、图标库或构建步骤。
- 继续使用 `internal/web/static/style.css` 作为 Web Console 的唯一共享样式入口，以 CSS 自定义属性和语义类形成小而稳定的样式接口。
- 模板逐页移除纯展示用途的内联样式。仅保留数据驱动且不能由静态类表达的值，例如漏洞分布条宽度。
- 不建立抽象组件系统或模板 DSL。复用当前 `base.html` 外壳和已有语义结构，仅在重复结构确实存在时增加模板片段或共享类。
- 交互逻辑继续放在现有小型 JS 文件中。行为 hook 优先使用稳定的 `data-*` 属性，避免依赖纯视觉类名。
- 独立导出报告继续生成单个自包含 HTML 文件。它复制一小组已批准的视觉令牌，不增加 CSS 生成器或跨包运行时依赖。
- 删除现有移动端页眉、侧栏遮罩、移动断点样式及对应脚本；增加明确的 1280px 桌面最小宽度。
- 使用系统字体栈，不下载或打包 SF Pro 等外部字体。
- 现有 SVG 图标继续复用并统一尺寸、线宽和颜色；移除作为控件装饰的 emoji。

## Testing Decisions

以下 seam 可在行为变化时使用 `tdd`：

1. 原生 JS 交互 seam：报告筛选、popover、活动筛选标签、复制反馈、分页跳转和吸顶操作区新增的状态行为。
2. Web HTTP seam：模板仍渲染现有字段、链接、表单名称和查询参数，所有路由及导出行为保持不变。
3. 独立报告 seam：给定相同 `ScanReport` 时，报告内容和导出成功行为保持不变。

纯 CSS、间距、颜色和阴影不建立脆弱的 DOM 快照测试，以 1440px 截图审批和浏览器人工检查为验收 seam。每个实施 ticket 运行相关 Go/JS 测试；最后运行全量测试、静态检查和桌面页面矩阵验收。

## Acceptance Criteria

- 所有控制台页面和独立 HTML 报告使用统一的浅色视觉语言。
- 1440px 下关键页面通过逐阶段截图审批。
- 1280px 下左侧导航、主要工具栏、表单和报告表格完整可用，无非预期页面级横向溢出。
- 所有现有路由、数据字段、表单名称、查询参数、下载格式和业务行为保持不变。
- 报告页的筛选、视图切换、导出、复制、分页和命令生成仍可用。
- 可交互元素具有 hover、focus-visible、disabled、loading、empty 和 error 等适用状态。
- 正文、次要文字、边框和状态色达到基本 WCAG AA 对比度要求。
- 不新增前端依赖、构建工具、运行时框架或远程字体。
- 现有 Go 与原生 JS 测试全部通过。

## Out of Scope

- 移动端或平板端适配。
- 深色模式或主题切换器。
- 完整品牌重塑、全新 Logo 或营销站设计。
- 新业务功能、新报告字段、新导出格式或后端数据模型变化。
- React、Vue、Svelte、Tailwind、UI 组件库、图标依赖或 CSS-in-JS。
- 对现有路由、权限、扫描流程和数据库结构的改造。

## Delivery Rule

按 `tickets/` 中的阻塞边一次实施一个 ready frontier ticket。若截图审批否决当前视觉方向，先更新本 spec、`product-design.md` 和受影响 ticket，再继续实施。
