# Web Console 持续演进与 Vue 迁移

**Status:** approved

## Problem Statement

Web Console 已完成第一轮视觉现代化，但新增功能继续堆叠在 Go 模板、全局 CSS 和原生 JavaScript 上。当前主要问题已经从“页面不统一”变成：高频流程步骤偏多、交互状态分散、键盘行为不完整、错误反馈打断操作，以及缺少全局暗色主题。

继续逐段添加原生脚本会扩大维护成本；一次性改写成 SPA 又会把稳定的 Go 路由、表单和服务端渲染一起置于风险中。本计划采用可持续的渐进迁移：保留 Go 的既有职责，以 Vue 3 接管真正需要客户端状态的页面区域，并在每个垂直切片完成后删除对应旧实现。

本计划替代 `modernize-web-console` 中“排除深色模式、前端框架和构建步骤”的历史决定，但不改写该已完成计划的验收记录。

## Outcomes

1. 建立 Vue 3、Vite 和 TypeScript 的可重复构建、类型检查和浏览器验证流程。
2. 提供 `system / light / dark` 全局主题，默认跟随系统并记住显式选择。
3. 缩短扫描创建、工作台验证、报告筛选和 Run 跟踪的高频路径。
4. 让主要操作可仅使用键盘完成，并提供稳定的焦点与非阻塞反馈。
5. 逐页收回原生 JavaScript；同一行为不长期维护 Vue 与旧脚本两套实现。
6. 建立长期缺陷修复与能力增强机制，而不是在本轮 UI 改造结束后停止演进。

## Approved Direction

- 允许保守调整默认交互：单一 Zone 自动选择、低频字段折叠、浏览器本地偏好记忆和创建后进入 Run 详情。
- 采用 Vue 3 + Vite + TypeScript。
- 采用渐进增强，不进行一次性 SPA 重写。
- Go 继续负责路由、权限、业务规则、持久化和初始 HTML。
- Vue 负责需要客户端状态的交互区域；原生 HTML 能满足的静态内容与普通表单不强制组件化。
- 暂不引入 Vue Router、Pinia、UI 组件库、CSS-in-JS 或新的图标库。
- 独立导出报告继续使用浅色、可打印的自包含样式，不读取浏览器本地主题。

## Two Delivery Lanes

### A. 缺陷修复

- 安全、数据丢失、崩溃和主流程阻断问题立即在当前所有者模块修复，不等待 Vue 迁移。
- 普通行为缺陷进入下一个可执行 ticket；若相关页面尚未迁移，修复仍须留下最小回归检查。
- 纯视觉瑕疵可并入即将到来的页面切片，但不能用“以后会迁移”推迟影响使用的问题。
- 修复应落在共享根因处；页面迁移后删除被替代的旧测试和旧逻辑。

### B. 优化与能力增强

- 本计划只批准主题、交互简化、可访问性和 Vue 迁移基础。
- 新业务能力单独建立 spec，再排入最近的页面切片；框架迁移本身不预先授权业务扩张。
- 每个阶段只迁移一个可独立发布的垂直切片，并以用户可观察结果验收。

## User Stories

1. 作为操作者，我可以快速填写常用扫描信息，而不必穿过大量低频参数。
2. 作为分析人员，我可以用更少的筛选和按钮完成候选验证与报告检索。
3. 作为长时间使用控制台的用户，我可以选择跟随系统、浅色或深色外观。
4. 作为键盘用户，我可以完成创建扫描、筛选、弹窗确认和查看状态等主要流程。
5. 作为维护者，我可以在带类型检查和清晰状态所有权的 Vue 模块中修改交互，而不必追踪分散的 DOM 副作用。
6. 作为发布者，我可以继续生成单个 Go 可执行文件，前端构建产物随二进制嵌入。

## Long-term Roadmap

| 阶段 | 用户结果 | 架构结果 | 退出条件 |
|---|---|---|---|
| 1. 构建与主题 | 全局三态主题可用 | Vue/Vite/TS 接入现有 Go 构建 | 本地、CI、发布包都包含确定性前端产物 |
| 2. 扫描创建 | 高频字段首屏可完成 | 第一个完整 Vue 垂直切片 | 旧表单联动脚本删除，Go 提交契约不变 |
| 3. 验证工作台 | 筛选和候选操作明显收敛 | 工作台状态归 Vue 所有 | 不再由内联样式和旧脚本维护同一状态 |
| 4. 报告阅读 | 搜索、筛选和视图切换更直接 | 报告交互迁入 Vue | 查询参数与导出语义保持兼容 |
| 5. Run 与工具 | 长任务反馈连续、可离开再返回 | 轮询与 dialog 状态统一 | 更新不抢焦点、不重置用户视图 |
| 6. 其余页面 | 项目、导入、知识库和配置交互一致 | 清理剩余通用旧脚本 | 原生脚本只保留无需框架的微小独立行为 |
| 7. 收口 | 发布与维护路径稳定 | 删除迁移脚手架和重复接口 | 全量审计、文档与回归门禁通过 |

路线图按退出条件推进，不承诺日历日期。每完成一个阶段，根据真实缺陷与使用反馈修订后续 ticket。

## Acceptance Criteria

- `make pr-check` 覆盖前端依赖安装后的类型检查、生产构建、Go 测试和现有浏览器 smoke。
- Go 构建和发布包嵌入生产前端产物，不在运行时依赖 Node 或外部 CDN。
- `system / light / dark` 在所有控制台页面一致生效，显式选择跨刷新保留；首次绘制不出现明显错误主题闪烁。
- 明暗主题分别满足调研文档中的文字、控件、焦点与状态对比度要求。
- 每个迁移切片保留现有路由、表单字段、查询参数和业务语义，除非其 ticket 明确批准变更。
- 同一交互在切片完成后只有一个状态所有者；被 Vue 替代的旧脚本与旧测试同步删除。
- 主要路径具备最小浏览器回归检查；非平凡纯逻辑保留聚焦测试。
- 每个 ticket 完成前执行代码审查，并用 1440px 明暗两套关键截图做视觉验收。

## Out of Scope

- 一次性 SPA 重写。
- 为迁移而重写稳定的 Go 业务逻辑或数据库模型。
- 移动端和平板端重新设计。
- 自定义主题色、密度编辑器、可拖拽列、全局命令面板和大量快捷键。
- 在没有已批准需求时增加新的扫描能力或报告格式。

## Delivery Rule

使用 `tickets/` 的阻塞边一次实施一个 ready frontier ticket。当前所有 ticket 均为 `proposed`；计划批准后只把 ticket 01 改为 `ready-for-agent`。实施中若发现 Vue/Go seam 或用户流程假设无效，先修订本 spec 与受影响 ticket，再继续。
